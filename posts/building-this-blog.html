Building This Blog for the Open Web

2026.07.107 min readTopic: architecture, deployability

这次改造里，最重要的决定不是重新配色，也不是继续加视觉特效，而是承认一件事：一个真正的博客首先要像博客，而不是像一个演示视频里的网页。

1. 原始版本的问题不只是“细节不够”

最初的版本已经有很强的气质，但它更接近单页概念稿。文章在浮层里、互动是本地假的、留言和订阅都不真实、很多入口无法分享。也就是说，问题不只是颜色和细节，而是产品结构本身还没有站稳。

2. 这次真正改变的是信息架构

一旦把首页、文章页、项目页和关于页拆开，很多问题自然就消失了。首页不必塞下所有内容，文章可以拥有独立 URL，项目也终于有了自己的入口。这时候视觉语言反而更容易统一，因为每一页都知道自己该承担什么职责。

  • 首页：建立气质，做文章导览。
  • 文章页：沉浸阅读，提供目录与进度。
  • 项目页：集中放作品，不和文章竞争首屏。
  • 关于页：解释创作方向与联系方式。

3. 去掉假功能，本身也是一种完成度提升

很多原型会靠“看起来很热闹”的交互撑起完成感，比如假访客数、假点赞、假订阅、假留言。它们在演示时好用，但上线后反而会削弱信任。与其把时间继续花在这些伪功能上，不如把它们替换成真实可达的链接、真实评论系统的预留位，或者暂时先不做。

4. 为什么静态优先更适合这个博客

对个人博客来说，静态优先不是退而求其次，而是一种非常强的默认选择。它部署简单、迁移成本低、依赖少、版本控制清晰。尤其你已经明确后续会放 GitHub，这种结构会比继续堆单页脚本可靠得多。

当内容是主角时，稳定性就是设计的一部分。

5. 这之后最值得做什么

下一步不是继续抠首页粒子，而是把内容数据和发布流程接起来：确定文章来源、决定评论方案、加入 feed、接好 GitHub Pages 自动部署。到那一步，这个博客就不再只是一个好看的方向，而会变成一个真正能长期写下去的站点。
